## 사용자/목적

로그인 사용자는 더 이상 필요 없는 자신의 카테고리를 삭제할 수 있어야 한다.

## 범위

- 사용자는 자신의 카테고리를 영구 삭제한다.
- 존재하지 않거나 다른 사용자의 카테고리 삭제 요청은 거절한다.
- 삭제 전 화면에서 확인 대화상자를 보여준다.
- 삭제 확인 대화상자는 카테고리에 묶였던 할 일이 미분류로 바뀐다는 설명을 보여준다.
- 삭제 요청 중에는 중복 제출을 막는다.
- 삭제 시 연결된 할 일을 유지하고 카테고리 연결만 해제하는 사용자 관점 검증은 REQ-026과 함께 추적한다.

## 표준 용어

- category.category
- category.delete
- category.notFound
- todo.task
- todo.categoryRef

## 제외 범위

- 카테고리 생성과 수정
- 삭제 시 영향받는 할 일 개수 표시
- 카테고리 삭제에 따른 할 일 목록 화면 검증

## 수용 기준

- (API) 본인의 카테고리를 영구 삭제할 수 있다
- (API) 존재하지 않는 카테고리를 삭제하려 하면 거절된다
- (API) 다른 사용자의 카테고리를 삭제하려 하면 거절된다
- (UI) 카테고리를 삭제하려고 하면 삭제를 확인받는 안내와 함께, 그 카테고리에 묶였던 할 일이 미분류로 바뀐다는 설명이 보인다
- (UI) 삭제를 확인하면 그 카테고리가 목록에서 사라진다
- (UI) 카테고리를 삭제하는 요청을 기다리는 동안 삭제 버튼은 다시 누를 수 없는 상태로 표시된다

## 의사결정 로그

- 결정일: 2026-05-21
  결정: 존재하지 않는 카테고리 삭제 요청도 수정과 동일하게 거절한다.
  이유: 동일 자원의 부재·타인 접근 정책을 통일해 클라이언트 오류 처리를 일관되게 한다.
  결정자: Tech Lead
  영향: 삭제 핸들러는 부재와 타인 접근을 동일한 분기로 처리한다.

- 결정일: 2026-05-21
  결정: 카테고리 삭제 시 연결된 본인 할 일의 카테고리 연결을 해제하고 할 일 자체는 유지한다.
  이유: 카테고리 정리만으로 할 일 데이터를 잃지 않게 한다.
  결정자: Product Owner, Tech Lead
  영향: 삭제 구현은 TodoService 연결 해제를 호출한다. 사용자 관점 AC는 REQ-026과 함께 추적한다.

- 결정일: 2026-05-30
  결정: 삭제 확인 대화상자는 영향받는 할 일 개수를 표시하지 않고, 묶였던 할 일이 미분류로 바뀐다는 일반 안내만 표시한다.
  이유: 카테고리 조회 응답에 카테고리별 할 일 건수가 없어 개수를 정확히 보여줄 수 없다.
  결정자: Tech Lead
  영향: 삭제 확인 대화상자는 건수 없이 미분류 전환 안내를 표시한다.
