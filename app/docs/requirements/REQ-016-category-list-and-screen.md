## 사용자/목적

로그인 사용자는 카테고리 관리 화면에 들어가 자신의 카테고리 목록을 정해진 순서로 볼 수 있어야 한다.

## 범위

- 사용자는 자신의 카테고리 목록을 `displayOrder` 오름차순, 동률이면 먼저 등록한 카테고리 우선으로 조회한다.
- 보호 영역에 `/categories` 경로의 카테고리 관리 화면을 둔다.
- 비인증 사용자가 카테고리 화면에 접근하면 로그인 화면으로 이동한다.
- 카테고리 화면 경로로 진입했다가 로그인하면 카테고리 화면으로 돌아온다.
- 화면은 보호 화면 앱 셸 안에서 동작하며, 할 일 화면과 카테고리 화면 사이를 오갈 수 있는 내비를 보여준다.
- 화면 목록은 한 묶음 20개 기반 가상 스크롤로 다음 묶음을 이어 보여준다.
- 카테고리가 없으면 빈 상태 안내를 보여준다.

## 수용 기준

- (API) 본인의 카테고리는 정해진 정렬 순서대로 보이며, 같은 순서면 먼저 등록한 카테고리가 위로 정렬되어 보인다
- (UI) `/categories` 경로에 접근하면 자신의 카테고리 목록이 보인다
- (UI) 카테고리 목록은 정해진 정렬 순서대로 보이며, 같은 순서면 먼저 등록한 카테고리가 위로 정렬되어 보인다
- (UI) 카테고리 목록의 각 항목은 이름과 색상을 함께 표시한다
- (UI) 카테고리가 한 묶음(20개)보다 많으면, 처음에는 첫 묶음의 카테고리까지만 보여주고 목록을 아래로 스크롤하면 다음 묶음의 카테고리를 이어서 보여준다
- (UI) 카테고리가 하나도 없으면 카테고리가 비어 있다는 안내가 보인다
- (UI) 카테고리 화면은 보호 화면 앱 셸 안에 보이고, 할 일 화면과 카테고리 화면을 오갈 수 있는 내비가 보인다
- (UI) 비인증 사용자가 카테고리 화면 경로에 접근하면 로그인 화면으로 이동한다
- (UI) 카테고리 화면 경로로 진입했다가 로그인하면 카테고리 화면으로 돌아온다
- (UI) 데스크톱 화면에서 카테고리 목록과 입력 영역의 주요 요소가 화면 밖으로 넘치지 않는다
- (UI) 카테고리 화면은 자동 접근성 검사에서 위반이 없어야 한다

## 의사결정 로그

- 결정일: 2026-05-21
  결정: 카테고리 목록은 `displayOrder` 오름차순, 동률 시 `id` 오름차순으로 반환한다.
  이유: 사용자가 지정한 정렬 의도를 우선하고 동률에서도 결정적 순서를 유지한다.
  결정자: Product Owner, Tech Lead
  영향: 목록 API와 화면은 같은 정렬 결과를 기준으로 검증한다.

- 결정일: 2026-05-30
  결정: 카테고리 화면은 보호 영역 전용 `/categories` route로 두고, 보호 화면 앱 셸에 할 일 화면과 카테고리 화면을 오가는 1차 내비를 도입한다.
  이유: 카테고리 화면을 독립적으로 검토하면서 이후 할 일 화면이 같은 앱 셸 내비를 재사용할 수 있게 한다.
  결정자: Product Owner, Tech Lead
  영향: `/categories`는 비인증 접근 시 로그인 화면으로 이동하고, 로그인 후 신뢰 이동 대상에 포함된다.

- 결정일: 2026-05-30
  결정: 카테고리 목록은 페이징 API 위에서 가상 스크롤로 동작한다.
  이유: 보이는 만큼만 렌더링하면서 스크롤로 전체 목록에 도달할 수 있게 한다.
  결정자: Product Owner, Tech Lead
  영향: 목록은 한 묶음 20개로 이어 불러오고 사용자에게 페이지 이동 컨트롤을 보여주지 않는다.
