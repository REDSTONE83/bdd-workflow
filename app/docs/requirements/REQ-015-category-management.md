## 사용자/목적

로그인 사용자는 자신의 할 일을 분류하기 위해 카테고리를 보고, 만들고, 고치고, 삭제할 수 있어야 한다. 신규 사용자는 가입 직후 기본 카테고리로 분류 흐름을 시작할 수 있어야 한다.

## 범위

- 본 카드는 카테고리 관리 기능의 상위 목적과 전체 사용자 여정만 소유한다.
- 목록 조회와 화면 진입은 REQ-016에서 관리한다.
- 카테고리 생성은 REQ-017에서 관리한다.
- 카테고리 수정은 REQ-018에서 관리한다.
- 카테고리 삭제는 REQ-019에서 관리한다.
- 신규 사용자 기본 카테고리 제공은 REQ-020에서 관리한다.

## 표준 용어

- category.category
- category.create
- category.list
- category.update
- category.delete
- category.default

## 제외 범위

- 하위 원자 요건의 상세 AC 복사
- 카테고리 수동 재정렬 UX
- 카테고리 검색과 필터
- 다른 사용자와의 카테고리 공유

## 수용 기준

- (E2E) 로그인 사용자는 카테고리 화면에서 카테고리를 확인하고 새 카테고리를 만든 뒤 수정하고 삭제할 수 있다

## 의사결정 로그

- 결정일: 2026-06-02
  결정: REQ-003과 REQ-014를 대체하는 카테고리 관리 상위 요건을 REQ-015로 발급한다.
  이유: 기존 REQ-003은 여러 생명주기 동작을 한 카드에 담았고, REQ-014는 화면 구현 슬라이스였다. 새 표준에서는 상위 요건과 하위 원자 요건으로 기능 소속을 표현한다.
  결정자: Product Owner, Tech Lead
  영향: REQ-003과 REQ-014는 `대체됨`으로 닫고, 상세 AC/Scenario/Test/코드 annotation은 REQ-016~REQ-020으로 이동한다.

- 결정일: 2026-06-02
  결정: 상위 요건에는 전체 카테고리 관리 여정의 `(E2E)` AC 하나만 둔다.
  이유: 상위 요건은 하위 요건 존재 여부를 검증하지 않고, 여러 원자 요건이 결합될 때 관찰되는 사용자 성과만 소유해야 한다.
  결정자: Product Owner, Tech Lead
  영향: 상세 생성/조회/수정/삭제/기본 시드 검증은 하위 원자 요건에서 관리한다.

- 결정일: 2026-06-05
  결정: 상위 요건의 `(E2E)` AC는 mock API가 아니라 실 백엔드와 실 프런트엔드를 함께 띄운 Playwright 통합 스모크 테스트가 커버한다.
  이유: 상위 요건은 개별 화면 상태보다 프런트엔드 라우팅, Vite proxy, HttpOnly Cookie, 백엔드 API가 결합된 사용자 성과를 확인해야 한다.
  결정자: REDSTONE
  영향: mock 기반 카테고리 E2E는 보조 회귀 테스트로 남기고, `REQ-015` 커버리지는 `tests/e2e/live`의 live spec이 소유한다.
