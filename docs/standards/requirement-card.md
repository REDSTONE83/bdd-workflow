# 요건 카드 표준

요건 카드는 `/docs/requirements/REQ-XXX-*.md` 파일로 관리한다. 카드는 사용자 요청 티켓이 아니라 제품 또는 하네스 명세의 원천이다. 사용자 요청은 여러 요건을 동시에 바꾸는 작업 묶음일 수 있지만, 승인과 완료 판정은 각 요건 카드의 수용 기준(AC) 단위로 계산한다.

카드에는 현재 보장해야 하는 최종 명세만 남긴다. 기능 변경이나 정책 전환이 있으면 변경 작업을 별도 REQ로 만들지 않고, canonical REQ의 `범위`, `제외 범위`, `수용 기준`, Scenario, Test를 최종 상태 기준으로 갱신한다. 변경 전 상태, 전환 작업 순서, 검토 이력은 Change Set과 `의사결정 로그`에만 남긴다.

카드는 사람이 5-15분 안에 검토할 수 있는 크기를 유지한다. 구현 순서 때문에 생긴 슬라이스 카드는 장기 명세로 보지 않고 `명세 역할: 구현 슬라이스`로 표시한 뒤 관련 원자 요건에 병합하거나 `대체됨`으로 닫는다.

## 헤더

모든 카드는 다음 헤더를 가진다.

```text
요건 ID: REQ-XXX
제목: 요건 제목
우선순위: 높음 | 중간 | 낮음
상태: 초안 | 검토중 | 승인 | 대체됨 | 폐기
요건 종류: 기능 | 비기능 | 통합 | 정책 | 하네스
명세 역할: 원자 요건 | 상위 요건 | 구현 슬라이스
대상 시스템: application | harness
제품 영역: auth | todo | category | platform | harness
품질 속성: none | accessibility | security | performance | compatibility | usability
검증 수준: acceptance | e2e | mixed | static
관련 요건: 없음 또는 REQ-001, REQ-002
대체 요건: 없음 또는 REQ-001
```

`제품 영역`의 `auth | todo | category | platform | harness`는 현재 사용하는 예시 키다. 닫힌 enum이 아니며, 새 영역이 필요하면 소문자/숫자/하이픈으로 된 안정적인 키를 쓴다.

`품질 속성`은 쉼표로 여러 개를 적을 수 있다. `none`은 다른 품질 속성과 함께 쓰지 않는다. `관련 요건`과 `대체 요건`은 존재하는 `REQ-XXX`만 적는다.

## 제목

제목은 장기 명세의 이름이다. Change Set의 작업명이나 구현 단계명을 제목으로 쓰지 않는다.

- 제목은 `대상 + 보장할 능력/정책/성과` 형태로 쓴다.
- 사용자가 체감하는 기능은 업무 능력이나 생명주기 동사로 쓴다. 예: `할 일 생성`, `카테고리 목록 조회`, `로그아웃 세션 종료`.
- 하네스 요건은 검사기, 리포트, 게이트, 표준, 추적 정책처럼 하네스가 소유하는 산출물과 보장 결과를 제목에 드러낸다. 예: `OpenAPI 기반 FE API 계약 검증`, `통합 하네스 게이트`.
- 상위 요건 제목은 제품 목표나 기능 묶음을 나타낸다. 하위 원자 요건 제목은 독립 승인 가능한 생명주기 동작, 정책 단위, 품질 보장 단위를 나타낸다.
- `백엔드`, `프런트엔드`, `API`, `화면`, `DB` 같은 구현 표면은 그 표면 자체가 명세 소유 대상일 때만 제목에 쓴다. 일반 기능은 구현 표면이 아니라 AC 마커, `검증 수준`, 코드/시나리오 연결로 표현한다.
- `도입`, `강화`, `전환`, `정리`, `후속`, `1차`, `Phase` 같은 작업 진행 표현은 Change Set 작업명으로 본다. REQ로 남길 때는 현재 보장할 결과 상태를 제목으로 쓴다.

예:

```text
권장:
- 제목: 할 일 생성
- 제목: 카테고리 목록 조회
- 제목: 로그인 화면
- 제목: JWT 기반 인증
- 제목: OpenAPI 기반 FE API 계약 검증
- 제목: 통합 하네스 게이트

검토 필요:
- 제목: 할 일 API 구현
- 제목: 할 일 화면 1차
- 제목: 인증 기능 개선
- 제목: JWT 기반 인증 전환
- 제목: 요구사항 카드 정리
```

## 분류 기준

`요건 종류`는 명세가 무엇을 소유하는지 나타낸다.

- `기능`: 사용자가 수행할 수 있어야 하는 업무 능력.
- `비기능`: 여러 기능 또는 특정 표면에 걸친 품질 보장.
- `통합`: 여러 기능/비기능이 결합된 시스템 여정 또는 최종 성과.
- `정책`: 여러 요건에 적용되는 현재 제품/기술 정책.
- `하네스`: 요구사항 추적, 검사기, 리포트, 게이트, 표준 문서 등 개발 하네스 자체.

`명세 역할`은 카드가 장기 명세인지 작업상 필요로 생긴 슬라이스인지 나타낸다.

- `원자 요건`: 독립적으로 승인하고 검증할 수 있는 기본 명세. 기본 선택이다.
- `상위 요건`: 여러 원자 요건을 묶는 제품 목표나 기능 단위. 같은 기능에 속한다는 사실을 표현할 때 별도 기능 ID를 만들지 않고 상위 REQ와 `관련 요건` 관계를 사용한다.
- `구현 슬라이스`: API 먼저, 화면 나중처럼 작업 순서 때문에 생긴 카드. 반드시 `관련 요건`을 적는다.

`대상 시스템`은 큰 경계만 표시한다. 사용자-facing 제품 동작, API, 화면, 데이터, 인증, 품질 요구는 `application`이다. 검사기, 리포트, 게이트, 표준 문서, 하네스 도구는 `harness`다.

큰 업무 카드를 사용자 목표와 생명주기 단위로 나눌 때는 다음처럼 표현한다.

- 상위 카드: `명세 역할: 상위 요건`. 기능 목적, 공통 범위, 통합 여정 AC를 가진다.
- 하위 카드: `명세 역할: 원자 요건`. 생성/조회/수정/삭제처럼 독립 승인 가능한 AC를 가진다.
- 하위 카드는 상위 카드와 같은 `제품 영역`을 쓰고, `관련 요건`에 상위 REQ를 적는다.
- 상위 카드는 하위 카드의 AC를 복사하지 않는다. 상위 카드의 AC는 여러 하위 요건이 함께 만족될 때 확인되는 사용자 여정이나 기능 성과만 둔다.
- 상위 카드는 얇은 기능 컨텍스트로만 유지한다. 구체 동작, 입력 검증, 예외 조건, 정렬/페이징, 화면 상태, 실행 테스트 연결은 하위 원자 요건으로 내려보낸다.
- 상위 카드의 AC는 기능 전체 성과를 검증할 수 있을 때만 둔다. 보통 `(E2E)` 또는 기능 성과 AC이며, 하위 요건 존재 확인용 구조 AC를 제품 상위 요건에 두지 않는다.
- 검증 가능한 AC가 없는 단순 묶음은 REQ로 만들지 않는다. 작업 범위면 Change Set에 두고, 설명 문서면 `docs/harness` 또는 `docs/standards`에 둔다.

## 분리와 병합

요건 분리는 구현 순서가 아니라 명세 의미와 검토 가능성을 기준으로 판단한다. 같은 제품 능력을 API, 화면, DB, 테스트 순서로 나눈 카드는 장기 REQ가 아니라 구현 슬라이스다.

다음 중 하나에 해당하면 별도 원자 요건으로 분리한다.

- 서로 다른 사용자 목표나 업무 생명주기 단계다. 예: 생성, 목록 조회, 상세 조회, 수정, 삭제, 완료 처리.
- 각 항목을 독립적으로 승인하거나 배포 영향도를 따로 판단할 수 있다.
- 정상/예외/상태 전이 규칙이 서로 달라 한 카드에서 AC가 서로 다른 정책을 섞는다.
- 품질 보장, 보안 정책, 호환성 정책처럼 여러 기능에 반복 적용되는 규칙이다.
- 적용 대상이나 검토자가 다르다. 예: 제품 기능, 하네스 게이트, 표준 문서, 공통 정책.
- 사람이 5-15분 안에 범위, 제외 범위, AC, 시나리오 영향을 검토하기 어렵다.

다음 경우에는 분리하지 않는다.

- API, UI, Entity, 테스트가 같은 사용자 능력을 구현하는 서로 다른 표면일 뿐이다.
- 한 업무 동작의 입력 검증, 저장, 응답, 화면 표시가 같은 승인 결과를 이루는 구성 요소다.
- 제목만 `화면`, `API`, `백엔드`, `프런트엔드`로 갈라져 있고 사용자가 얻는 결과가 같다.

상위 요건은 단순 폴더나 태그가 아니다. 여러 원자 요건이 함께 만족될 때 확인되는 제품 목표, 통합 여정, 최종 성과 AC가 있을 때만 만든다. 검증 가능한 AC가 없고 같은 기능에 속한다는 사실만 표현하려면 새 REQ를 만들지 않고 기존 상위 REQ의 `관련 요건` 관계나 Change Set 설명을 사용한다.

정책/비기능/하네스/통합 요건은 다음처럼 판단한다.

- 여러 기능에 공통 적용되는 규칙은 `요건 종류: 정책` 또는 `비기능`으로 분리한다.
- 검사기, 리포트, 게이트, 표준, 추적표 생성처럼 하네스가 직접 보장해야 하는 동작은 `요건 종류: 하네스`, `대상 시스템: harness`로 둔다.
- 기능 변경이나 정책 전환은 별도 전환 REQ로 남기지 않는다. 최종 기능/정책을 canonical REQ의 `범위`, `제외 범위`, `수용 기준`에 반영하고, 전환 작업과 변경 전 상태는 Change Set에 둔다.
- 기존 명세 구조, 정책, 추적 방식, 기술 기반, 데이터, 운영 상태를 바꾸는 작업이 필요해도 REQ는 작업 절차가 아니라 전환 후 보장할 최종 상태를 소유한다. 실행 절차, cutover, rollback, 데이터 보정 작업은 Change Set의 작업 범위와 검증 명령으로 관리한다.
- 사용자가 최종 시스템에서 여러 기능/비기능이 결합된 여정을 검토해야 하면 `요건 종류: 통합`, `검증 수준: e2e|mixed`로 둔다.

큰 업무 카드를 나눌 때 기존 큰 카드는 상위 요건으로 재사용하지 않는다. 상위 요건과 하위 원자 요건을 모두 다음 사용 가능한 `REQ-XXX`로 새로 발급하고, 기존 큰 카드는 `상태: 대체됨`, `대체 요건: REQ-XXX`로 닫는다. 이때 `대체 요건`은 신규 상위 REQ 하나만 가리키고, 하위 원자 요건은 신규 상위 REQ의 `관련 요건` 관계로 추적한다. 기존 ID를 여러 새 의미에 재사용하지 않는다.

신규 REQ ID는 현재 사용 중인 최대 번호 다음부터 순차 발급한다. 도메인별 번호 범위나 `REQ-002A` 같은 파생 ID를 만들지 않는다.

예:

```text
REQ-002 개인별 할 일 관리
상태: 대체됨
대체 요건: REQ-015

REQ-015 개인별 할 일 관리
명세 역할: 상위 요건
제품 영역: todo

REQ-016 할 일 생성
명세 역할: 원자 요건
제품 영역: todo
관련 요건: REQ-015

REQ-017 할 일 목록 조회
명세 역할: 원자 요건
제품 영역: todo
관련 요건: REQ-015
```

구현 순서 때문에 갈라진 카드는 canonical REQ로 병합한다. API 카드와 화면 카드처럼 제품 능력은 하나인데 구현 표면만 다른 경우가 여기에 해당한다. 병합된 카드는 `상태: 대체됨`, `대체 요건: REQ-XXX`로 닫는다. `대체 요건`은 canonical entry point 하나만 가리킨다. 병합 카드의 상세 AC, 시나리오, 의사결정 로그, 테스트 연결은 필요하면 여러 하위 원자 요건으로 나누어 옮길 수 있다.

분리 또는 병합으로 범위나 AC가 바뀐 카드는 기존 `승인`을 유지하지 않는다. 신규 상위/원자 카드와 병합 대상 canonical 카드는 `검토중`으로 시작하거나 되돌리고, AC/Scenario/Test/Report 정합성이 확인된 뒤 다시 `승인`으로 올린다. `대체됨` 또는 `폐기` 카드는 완료 판정 대상에서는 빠지지만 카드 구조 검증은 계속 받는다.

기존 카드의 상세 AC, Scenario, Test, 코드 annotation을 새 원자 요건으로 옮길 때는 원자 요건 하나씩 진행한다. 한 원자 REQ의 카드, `.feature`, 실행 테스트, 코드/FE metadata 참조를 함께 옮긴 뒤 단일 카드 trace로 확인하고 다음 원자 요건으로 넘어간다.

## 필수 섹션

- `## 사용자/목적`
- `## 범위`
- `## 표준 용어`
- `## 제외 범위`
- `## 수용 기준`
- `## 의사결정 로그`
- `## BDD 테스트 리뷰`
- `## 열린 질문`

새 카드는 `docs/templates/requirement-card.md`를 복사해 시작한다.

## 수용 기준

모든 수용 기준은 bullet 시작 직후에 검증 채널 마커를 가진다. 마커가 없는 AC는 카드 구조 오류다.

```text
- (API) 유효한 가입 정보이면 계정이 생성된다
- (UI) 회원 가입 화면은 필수 입력 오류를 화면에 표시한다
- (E2E) 신규 사용자는 가입 후 로그인하고 첫 할 일을 목록에서 확인할 수 있다
- (STATIC) 요건 카드 검사 결과는 새 카드 스키마 위반을 오류로 보고한다
```

마커 의미:

- `API`: 백엔드 Acceptance Test 커버가 필요하다.
- `UI`: Playwright FE BDD 테스트 커버가 필요하다.
- `E2E`: Playwright 기반 사용자 여정 테스트 커버가 필요하다.
- `STATIC`: 하네스/정적 검사/리포트 검증 테스트 커버가 필요하다. 테스트 위치는 백엔드 Acceptance Test 또는 FE BDD 테스트 어느 쪽이든 가능하다.

마커는 카드 표시 정보일 뿐 테스트 식별자는 아니다. 같은 AC를 가리키는 `@Covers`, FE BDD `Covers`, `.feature` Scenario `Covers:` 값에는 마커를 포함하지 않는다.

`검증 수준`과 AC 마커 집합은 서로 모순되지 않아야 한다.

- `검증 수준: acceptance`: 단일 제품 표면의 인수 검증이다. AC 마커는 `(API)` 또는 `(UI)` 중 한 종류만 가진다.
- `검증 수준: e2e`: 최종 사용자 여정 검증이다. AC 마커는 `(E2E)`만 가진다.
- `검증 수준: mixed`: 둘 이상의 검증 채널이 섞인 카드다. 예: `(API)` + `(UI)`, `(API)` + `(UI)` + `(E2E)`.
- `검증 수준: static`: 하네스/정적 검사/리포트 검증 카드다. AC 마커는 `(STATIC)`만 가진다.

AC는 사용자/PO/QA가 승인할 수 있는 관계자 언어로 적는다. 구현 방식이 아니라 관찰 가능한 결과여야 한다. `null`, JSON 키, HTTP 상태 코드, 오류 코드는 제품 자체가 외부 API 계약이고 관계자가 그 값을 직접 의존하는 경우가 아니면 AC에 쓰지 않는다.

## 시나리오

시나리오 원본은 `docs/scenarios/REQ-XXX-*.feature`에 둔다. 카드에는 시나리오 본문을 복제하지 않고 `BDD 테스트 리뷰` 섹션에 문서 링크와 승인 이력만 남긴다.

통합 요건은 하위 요건 AC를 복사하지 않는다. 통합 요건의 AC는 시스템 여정 또는 교차 정책의 결과만 소유한다. 하위 요건은 `관련 요건`으로 연결하고, 변경 영향은 의사결정 로그 또는 별도 영향 검토 이력으로 남긴다.

## 관련/대체 요건

- `관련 요건`: 본 카드가 참조하거나 함께 검토해야 하는 요건. 구현 슬라이스, 정책, 비기능, 통합 요건은 보통 관련 요건을 가진다.
- `대체 요건`: 상태가 `대체됨`인 카드가 어떤 canonical 요건으로 닫혔는지 나타낸다.

`상태: 대체됨`이면 `대체 요건`이 필수다. `상태: 폐기`는 제품 명세로 채택하지 않기로 한 카드에만 사용한다.

## 열린 질문 / 의사결정 로그

요건 작성 중 사용자에게 던졌으나 아직 확정되지 않은 질문은 `열린 질문`에만 둔다. 답변이 오면 다음 위치 중 하나로 반영하고 열린 질문에서 제거한다.

- 동작 자체가 바뀌면 `범위` 또는 `제외 범위`
- 검증 가능한 결과면 `수용 기준`
- 정책 선택과 근거를 남겨야 하면 `의사결정 로그`

의사결정 로그는 다음 항목을 가진다.

- 결정일
- 결정
- 이유
- 결정자
- 영향

결정이 AC를 바꾸면 카드 AC, `.feature` `Covers:`, 실행 테스트의 `@Covers` 또는 FE `Covers`를 함께 갱신한다. 카드를 `승인`으로 올리려면 열린 질문이 비어 있어야 한다.

## BDD 테스트 리뷰

초안 단계에서는 `미완료`로 둘 수 있다. Skeleton 단계에서는 검증 설계와 계약 골격 승인 결과를 남긴다. 수용 기준과 실행 테스트가 리뷰되기 전에는 요건 카드 상태를 `승인`으로 바꾸지 않는다.

Skeleton 승인 이력은 필요 항목만 남긴다.

- `검증 설계`: 카드 AC와 `.feature` Scenario `Covers:` 매핑 요약.
- `API Skeleton`: API/DTO/Service/Repository 계약 골격.
- `DB Skeleton`: Entity/컬럼/관계와 `previewSchema` 확인 결과.
- `화면/라우팅 Skeleton`: 화면 이름, route, 접근 권한, 주요 상태, Storybook story 위치.
- `검사기 Skeleton`: 하네스 collector/validator/reporter/gate 변경.
- `추적 정책`: AC 마커, RED/GREEN/BLUE 판정, 관련 요건 영향.
- `검증`: 실행한 명령과 결과.
- `승인자`
- `Skeleton 결과`: `승인` / `승인 대기` / `미완료`

## 금지 사항

- API ID, 화면 ID, 시나리오 ID 같은 별도 사람이 관리하는 ID를 만들지 않는다.
- 카드에 API 목록, 테스트 메서드 목록, 화면 목록을 수기로 유지하지 않는다. 자동 리포트에서 확인한다.
- 사용자 요청 하나를 무조건 새 REQ로 만들지 않는다. 먼저 기존 원자 요건 수정, 통합 요건, 정책 요건, 비기능 요건, 구현 슬라이스 중 무엇인지 판단한다.
- 기능 변경, 정책 전환, 기술 전환을 `... 도입`, `... 전환`, `... 강화` 같은 별도 REQ로 남기지 않는다. 최종 명세는 canonical REQ에 남기고 작업 이력은 Change Set에 둔다.

## 자동 검증

`validateRequirementCards`, `validateHarness`, `validateRequirementCard*`는 새 카드 스키마만 유효한 것으로 본다. 통합 게이트 `gate.mjs`는 REQ-010 계약에 따라 TRACE/CARD/REF/TRC/BE/FE/SCN/TRM 카테고리를 한 번에 판정하며, 용어 findings의 `strictSeverity=error`는 TRM 실패로 차단한다.

- 헤더 필수값 누락/허용값 위반
- `품질 속성: none`과 다른 속성 혼용
- 관련/대체 요건의 존재 여부
- `상태: 대체됨`인데 `대체 요건` 누락
- `명세 역할: 구현 슬라이스`인데 `관련 요건` 누락
- `요건 종류: 통합`인데 `검증 수준`이 `e2e|mixed`가 아니거나 `(E2E)` AC가 없음
- `검증 수준`과 AC 마커 집합 불일치
- 수용 기준 비어 있음, 중복, 마커 누락(`CARD-AC-MARKER-MISSING`), 마커 허용값 위반(`CARD-AC-MARKER-INVALID`)
- 표준 용어 형식/등록/중복
- 승인 카드의 열린 질문 또는 BDD 테스트 리뷰 미승인

요건 카드 스키마 리포트는 `build/harness/reports/requirement-schema-report.md`와 `.json`에 생성된다.
