# Change Set: 요건 카드 스키마 전환과 REQ 정리 준비

상태: 진행중
요청일: 2026-06-01
변경 유형: 하네스 개선, 마이그레이션, 분리, 병합, 대체
영향 요건: REQ-001, REQ-002, REQ-003, REQ-004, REQ-005, REQ-006, REQ-007, REQ-008, REQ-009, REQ-010, REQ-011, REQ-012, REQ-013, REQ-014, REQ-015, REQ-016, REQ-017, REQ-018, REQ-019, REQ-020, REQ-021, REQ-022, REQ-023, REQ-024, REQ-025, REQ-026, REQ-027
논의 상태: 없음

## 요청 요약

- 사용자 요청을 Change Set으로 보고, REQ는 명세 원천, AC는 기계 검증 기준, Scenario는 업무 검토 언어, Test는 실행 검증, Report는 생성 산출물로 정렬한다.
- 새 요건 카드 스키마와 하네스 리포트를 먼저 도입한 뒤, 기존 REQ 재분류는 사용자와 적정 단위를 논의한 후 진행한다.

## 작업 범위

- Change Set 원본 위치, 템플릿, 작성 기준을 문서화한다.
- Change Set 인덱스와 리포트를 생성해 영향 REQ의 카드 스키마 상태와 추적 상태를 한 화면에서 확인한다.
- 기존 REQ 재분류는 사용자와 적정 단위를 확정한 뒤 원자 요건 단위로 진행한다.
- REQ-013은 REQ-001로 병합한다.
- REQ-003/REQ-014는 REQ-015 상위 요건과 REQ-016~REQ-020 원자 요건으로 분리한다.
- REQ-002는 REQ-021 상위 요건과 REQ-022~REQ-027 원자 요건으로 분리한다.
- REQ-004는 기존 ID를 유지하고 정책 원자 요건 카드로 새 스키마에 맞춘다.
- REQ-005는 기존 ID를 유지하고 application/platform 원자 요건으로 새 스키마에 맞춘다.
- REQ-006은 기존 ID를 유지하고 harness 원자 요건으로 새 스키마에 맞춘다.
- REQ-007~REQ-010은 기존 ID를 유지하고 harness 원자 요건으로 새 스키마에 맞춘다.
- REQ-011은 기존 ID를 유지하고 application/auth 원자 요건으로 새 스키마에 맞춘다.
- REQ-012는 기존 ID를 유지하고 harness 원자 요건으로 새 스키마에 맞춘다.
- REQ-024의 완료 상태 변경 AC는 REQ-027로 분리한다.

## 제외 범위

- 아직 논의하지 않은 기존 REQ의 원자/통합/정책/하네스 재분류 확정
- 아직 확정하지 않은 기존 REQ 카드 본문 마이그레이션
- Change Set을 영구 명세 ID 체계로 승격

## 완료 조건

- Change Set이 명세 원천이 아니라 작업 범위임이 문서화된다.
- `build/harness/indexes/change-sets.index.json`이 생성된다.
- `build/harness/reports/change-set-report.md`와 `.json`이 생성된다.
- 현재 Change Set 리포트에서 영향 REQ의 카드 스키마 상태와 RED/GREEN/BLUE 상태를 확인할 수 있다.
- REQ 재분류 전에 사용자와 적정 단위 기준을 논의한다.

## 결정 로그

- 2026-06-01: Change Set은 사용자 요청을 처리하기 위해 함께 바꾸는 REQ 집합으로 두고, 별도 사람이 관리하는 ID를 만들지 않는다.
- 2026-06-01: 기존 REQ 재분류는 Change Set 리포트가 준비된 뒤 사용자와 적정 단위를 논의하고 진행한다.
- 2026-06-01: 구현 순서 때문에 API/화면/검사기/리포트 표면별로 갈라진 카드는 독립 REQ로 유지하지 않고 canonical REQ로 병합한다.
- 2026-06-01: 큰 업무 카드는 사용자 목표와 생명주기 단위로 원자 요건을 분리하고, 같은 기능에 포함된다는 사실은 별도 기능 ID가 아니라 `명세 역할: 상위 요건`, 동일 `제품 영역`, 하위 카드의 `관련 요건`으로 표현한다.
- 2026-06-01: 상위 요건은 얇은 기능 컨텍스트로만 유지한다. 기존 큰 REQ의 상세 AC, 시나리오, 테스트 연결은 하위 원자 요건으로 이동하고, 상위 요건은 공통 정책과 기능 전체 성과만 소유한다.
- 2026-06-01: 상위 요건은 기능 전체 성과를 검증할 수 있는 `(E2E)` 또는 기능 성과 AC가 있을 때만 AC를 둔다. 검증 가능한 전체 성과가 없으면 상위 REQ로 유지하지 않고 Change Set 또는 설명 문서로 둔다.
- 2026-06-01: 기존 큰 REQ는 상위 요건으로 재사용하지 않는다. 상위 요건과 하위 원자 요건을 모두 신규 REQ로 발급하고, 기존 큰 REQ는 `상태: 대체됨`과 `대체 요건`으로 닫는다.
- 2026-06-01: 기존 큰 REQ의 `대체 요건`은 신규 상위 REQ 하나만 가리킨다. 신규 하위 원자 요건은 신규 상위 REQ의 `관련 요건` 관계로 추적한다.
- 2026-06-01: 구현 표면 때문에 갈라진 카드의 `대체 요건`도 canonical entry point 하나만 가리킨다. 상세 AC, Scenario, Test는 필요하면 여러 하위 원자 요건으로 나누어 이동한다.
- 2026-06-01: 신규 REQ ID는 현재 사용 중인 최대 번호 다음부터 순차 발급한다. 현재 마이그레이션 기준 신규 요건은 `REQ-015`부터 시작한다.
- 2026-06-02: 신규 상위/원자 REQ는 모두 `검토중`으로 시작한다. 기존 승인 상태를 승계하지 않고, AC/Scenario/Test/Report 정합성 확인 후 다시 `승인`으로 올린다.
- 2026-06-02: 코드/시나리오/테스트 참조 이동은 원자 요건 하나씩 진행한다. 각 원자 REQ마다 카드, Scenario, Test, 코드 annotation을 옮기고 단일 카드 trace로 확인한다.
- 2026-06-02: `REQ-013 이메일 회원 가입 화면`은 `REQ-001 이메일 회원 가입`으로 병합했다. `REQ-001`은 API/화면 canonical 원자 요건으로 `검토중` 상태가 되었고, `REQ-013`은 `대체됨` + `대체 요건: REQ-001`로 닫았다.
- 2026-06-02: `REQ-003 할 일 카테고리 관리`와 `REQ-014 할 일 카테고리 관리 화면`은 `REQ-015 카테고리 관리` 상위 요건으로 대체했다. 상세 명세는 `REQ-016 카테고리 목록 조회`, `REQ-017 카테고리 생성`, `REQ-018 카테고리 수정`, `REQ-019 카테고리 삭제`, `REQ-020 신규 사용자 기본 카테고리 제공`으로 분리했다.
- 2026-06-02: 카테고리 관리 상위 요건은 하위 AC를 복사하지 않고 전체 카테고리 관리 여정 `(E2E)` AC 하나만 소유한다. 하위 원자 요건은 API/UI AC를 자기 생명주기 단위로 소유한다.
- 2026-06-02: `REQ-002 개인별 할 일 관리`는 `REQ-021 개인별 할 일 관리` 상위 요건으로 대체했다. 상세 명세는 `REQ-022 할 일 생성`, `REQ-023 할 일 목록 조회`, `REQ-024 할 일 수정`, `REQ-025 할 일 삭제`, `REQ-026 카테고리 삭제 시 할 일 연결 해제`, `REQ-027 할 일 완료 상태 변경`으로 분리했다.
- 2026-06-02: 할 일 관리 상위 요건은 하위 AC를 복사하지 않고 API lifecycle 기능 성과 AC 하나만 소유한다. 하위 원자 요건은 API AC를 생명주기 단위로 소유한다.
- 2026-06-02: `REQ-004 JWT 기반 인증`은 새 ID로 분리하지 않고 기존 ID를 유지했다. JWT 기반 인증은 여러 보호 API에 공통 적용되는 정책 원자 요건으로 분류하고, 모든 AC를 `(API)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-005 애플리케이션 기본 앱 셸`은 새 ID로 분리하지 않고 기존 ID를 유지했다. 애플리케이션 기본 앱 셸과 공개 진입 화면 smoke 검증을 소유하는 platform 원자 요건으로 분류하고, 모든 AC를 `(UI)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-006 OpenAPI 기반 FE API 계약 검증`은 새 ID로 분리하지 않고 기존 ID를 유지했다. OpenAPI 계약 인덱스와 FE API 정적 검사 룰을 소유하는 harness 원자 요건으로 분류하고, 모든 AC를 `(STATIC)` 검증 채널로 전환했다.
- 2026-06-02: 이미 이관된 카드 제목을 새 제목 표준으로 점검했다. `REQ-016`은 구현 표면 표현을 제거해 `카테고리 목록 조회`로, `REQ-006`은 하네스 보장 결과 중심의 `OpenAPI 기반 FE API 계약 검증`으로 정리했다. `REQ-004`는 작업 행위인 `전환`을 제거해 최종 정책 상태인 `JWT 기반 인증`으로 정리했고, `REQ-005`는 구현 기반 표현을 제거해 `애플리케이션 기본 앱 셸`로 정리했다. `REQ-024`는 `할 일 수정`으로 축소하고 완료 상태 변경을 별도 원자 요건으로 분리하기로 했다.
- 2026-06-02: `REQ-007 OpenAPI 기반 FE API 클라이언트 생성`은 기존 ID를 유지하고 harness 원자 요건으로 분류했다. API client 생성 파이프라인과 OpenAPI sha256 메타파일 검증을 소유하며, 모든 AC를 `(STATIC)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-008 FE API 계약 오류 검사`는 기존 ID를 유지하고 harness 원자 요건으로 분류했다. FE API drift, generated client metadata, API 경계 밖 직접 fetch 오류 검사를 소유하며, 모든 AC를 `(STATIC)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-009 Gherkin 시나리오 SCN-* 구조 검사`는 기존 ID를 유지하고 harness 원자 요건으로 분류했다. Layer 2 전환 작업명이 아니라 최종 SCN-* 구조 검사 보장 상태를 제목과 범위에 남기고, 모든 AC를 `(STATIC)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-010 통합 하네스 게이트`는 기존 ID를 유지하고 harness 원자 요건으로 분류했다. `도입` 작업 표현을 제거하고 8개 게이트 카테고리 보장 상태를 소유하게 했으며, 모든 AC를 `(STATIC)` 검증 채널로 전환했다.
- 2026-06-02: `REQ-011 이메일·비밀번호 로그인과 로그아웃`은 기존 ID를 유지하고 application/auth 원자 요건으로 분류했다. API 계약, UI 상태, 로그인·보호 화면·로그아웃 사용자 여정을 한 canonical 기능 카드에 유지하고, AC 검증 채널을 `(API)`, `(UI)`, `(E2E)`로 전환했다.
- 2026-06-02: `REQ-012 AC 단위 검증 채널 마커와 하네스 게이트`는 기존 ID를 유지하고 harness 원자 요건으로 분류했다. 구 `(BE)/(FE)/(FS)` 및 fallback 설명을 제거하고 현행 `(API)/(UI)/(E2E)/(STATIC)` 필수 마커 정책과 누락/오류 차단 규칙을 최종 명세로 정리했다.
- 2026-06-02: `REQ-024`는 `할 일 수정` 원자 요건으로 축소하고, 완료 상태 변경은 신규 `REQ-027 할 일 완료 상태 변경` 원자 요건으로 분리했다.
