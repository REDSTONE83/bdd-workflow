# Change Set: 기능별 UI/API 요건 배치 기준 보완

상태: 완료
요청일: 2026-06-06
변경 유형: 하네스 개선, 수정
영향 요건: REQ-021, REQ-022, REQ-023, REQ-024, REQ-025, REQ-027
논의 상태: 없음

## 요청 요약

- 같은 기능의 API 수용 기준과 화면 수용 기준을 별도 화면 카드로 분리하지 않고 원자 기능 카드에 함께 두는 기준을 명확히 한다.
- 구현 순서상 화면 슬라이스 카드가 생겼을 때 canonical 원자 요건으로 AC, Scenario, Test metadata를 이관하는 절차를 문서화한다.

## 작업 범위

- 요건 작성 가이드에 기능별 UI/API co-location 원칙을 추가한다.
- 요건 카드 표준에 화면/구현 표면 카드의 임시성과 이관 기준을 보강한다.
- 할 일 관리 화면 구조 정리를 이 기준의 적용 사례로 남긴다.

## 제외 범위

- 하네스 validator의 신규 룰 추가.
- 새 하네스 요건 카드 발급.

## 완료 조건

- 작성 가이드와 카드 표준이 기능별 UI/API 배치 기준을 설명한다.
- 할 일 관리 UI AC가 `REQ-022`, `REQ-023`, `REQ-024`, `REQ-025`, `REQ-027`로 이관된다.

## 검증 명령

- `npm run harness:validate`
- `npm run app:validate`

## 결정 로그

- 기능 자체가 하나이면 API와 UI가 다른 구현 표면이어도 원자 기능 카드에 함께 둔다.
- 화면 전용 카드는 독립 제품 능력이 아니라 구현 순서상 생긴 슬라이스일 때 canonical 원자 요건으로 이관한다.
- 상위 요건은 하위 AC를 복사하지 않고 실제 사용자 전체 여정만 `(E2E)` AC로 둔다.
