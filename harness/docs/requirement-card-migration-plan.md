# 요건 카드 스키마 마이그레이션 계획

이 문서는 2026-06 요건 카드 전환을 설명하는 1회성 마이그레이션 노트다. 작업 범위와 진행 상태의 원천은 `docs/change-sets/2026-06-01-requirement-card-harness-migration.md`이며, 이 문서는 기준, 단계, 판정 초안을 보조 기록으로만 남긴다. 향후 사용자 요청 단위 작업은 별도 계획 문서가 아니라 Change Set으로 관리한다.

## 목표

요건 카드를 사용자 요청 티켓이 아니라 제품/하네스 명세의 원천으로 재정렬한다. 하네스는 새 카드 스키마만 추출하고, 새 기준에 맞지 않는 카드는 일반 `CARD-*` 오류와 `requirement-schema-report`로 드러낸다. 이전 형식을 별도 deprecated 케이스로 식별하지 않는다.

## 완료 조건

- `build/harness/reports/change-set-report.md`에서 이 마이그레이션 Change Set과 영향 REQ 상태를 확인할 수 있다.
- `build/harness/findings/requirement-cards.findings.json`의 schema 관련 error가 0건이다.
- `build/harness/reports/requirement-schema-report.md`에서 incomplete 카드가 0건이다.
- 모든 AC가 `(API)`, `(UI)`, `(E2E)`, `(STATIC)` 중 하나의 마커를 가진다.
- `npm run validate`에서 RED가 0건이다.
- `상태: 대체됨` 카드는 `대체 요건`을 가지고, trace 상태에서 `INACTIVE`로만 남는다.

## 단계

0. Change Set 도입
   - 사용자 요청을 Change Set으로 기록한다.
   - Change Set은 명세 원천이 아니라 작업 범위로만 사용한다.
   - 기존 REQ 재분류는 Change Set의 `열린 논의`에 남기고, 사용자와 적정 단위를 확정한 뒤 진행한다.
   - `change-set-report`에서 영향 REQ의 카드 스키마 상태와 RED/GREEN/BLUE 상태를 확인한다.

1. 하네스 스키마 전환
   - `index-requirements.mjs`가 새 헤더만 추출한다.
   - `validate-requirement-cards.mjs`가 새 스키마 필수값과 AC 마커를 검증한다.
   - `evaluate-trace-state.mjs`가 카드 단위 back/front fallback 없이 AC 마커만으로 검증 채널을 계산한다.
   - `render-requirement-schema-report.mjs`가 카드별 스키마 적합성 리포트를 생성한다.

2. REQ 적정 단위 논의
   - API/화면/테스트 위치가 아니라 사용자가 승인할 수 있는 업무 능력, 정책, 품질 보장, 하네스 능력을 기준으로 나눈다.
   - 여러 사용자 목표, 여러 생명주기 단계, 서로 다른 정책 결정이 한 카드에 섞이면 사용자 목표와 생명주기 단위로 분리한다.
   - 같은 기능에 속하는 원자 요건은 별도 기능 ID를 만들지 않고 `명세 역할: 상위 요건`인 REQ와 연결한다.
   - 상위 요건은 기능의 목적, 공통 범위, 통합 여정 AC를 소유하고, 하위 원자 요건은 생성/조회/수정/삭제처럼 독립 승인 가능한 AC를 소유한다.
   - 상위 요건은 얇은 기능 컨텍스트로만 유지하고, 기존 큰 REQ의 상세 AC/시나리오/테스트 연결은 하위 원자 요건으로 이동한다.
   - 상위 요건의 AC는 기능 전체 성과를 검증할 수 있을 때만 둔다. 검증 가능한 전체 성과가 없으면 상위 REQ로 유지하지 않는다.
   - 하위 원자 요건은 상위 요건과 같은 `제품 영역`을 쓰고 `관련 요건`에 상위 REQ를 적는다.
   - 기존 큰 REQ는 상위 요건으로 재사용하지 않는다. 신규 상위 REQ와 신규 원자 REQ를 발급하고, 기존 큰 REQ는 신규 상위 REQ 하나를 `대체 요건`으로 적어 `대체됨`으로 닫는다.
   - API 카드와 화면 카드처럼 구현 순서 때문에 분리된 카드는 하나의 canonical REQ로 병합한다.
   - 병합되는 카드는 canonical 카드로 AC/시나리오/의사결정 로그를 이동한 뒤 `상태: 대체됨`, `대체 요건: REQ-XXX`로 닫는다.
   - 통합 요건은 하위 AC를 복사하지 않고 최종 시스템 여정의 성과만 소유하게 한다.

3. 카드 1차 분류
   - 모든 카드에 `요건 종류`, `명세 역할`, `대상 시스템`, `제품 영역`, `품질 속성`, `검증 수준`, `관련 요건`, `대체 요건`을 채운다.
   - 기존 구현 순서 때문에 생긴 카드는 `명세 역할: 구현 슬라이스`로 표시하고 관련 canonical 요건을 연결한다.
   - 신규 REQ는 현재 최대 번호 다음부터 순차 발급한다. `REQ-015`~`REQ-020`은 카테고리 관리 재분류에, `REQ-021`~`REQ-027`은 개인별 할 일 관리 재분류에 사용했으며, 다음 신규 요건은 다음 미사용 번호부터 시작한다.
   - 신규 상위/원자 REQ는 모두 `검토중`으로 시작한다. 기존 승인 상태를 승계하지 않는다.
   - 코드/시나리오/테스트 참조 이동은 원자 요건 하나씩 진행하고 단일 카드 trace로 확인한다.

4. AC 마커 전환
   - API/서비스/DB 검증은 `(API)`.
   - 화면/라우팅/컴포넌트 검증은 `(UI)`.
   - 여러 기능을 관통하는 사용자 여정은 `(E2E)`.
   - 하네스 도구, 리포트, 정적 검사 검증은 `(STATIC)`.
   - `@Covers`, FE `Covers`, `.feature` `Covers:` 문장에는 마커를 넣지 않는다.

5. canonical 요건 정리
   - API 카드와 화면 카드가 구현 순서 때문에 분리된 경우 하나의 원자 요건으로 합친다.
   - 기존 슬라이스 카드는 canonical entry point 하나를 `대체 요건`으로 적어 `상태: 대체됨`으로 닫는다.
   - 슬라이스 카드의 상세 AC/시나리오/테스트 연결은 필요하면 여러 하위 원자 요건으로 나누어 이동한다.
   - 공통 정책 카드는 제품 기능 카드에 병합하지 않고 `관련 요건`으로 영향 관계를 유지한다. 정책 전환 작업은 Change Set에 두고, 정책 REQ에는 최종 정책만 남긴다.

6. 통합/비기능 요건 정리
   - 최종 시스템 여정은 `요건 종류: 통합`, `검증 수준: e2e | mixed`로 둔다.
   - 통합 요건은 하위 요건 AC를 복사하지 않고 시스템 성과만 AC로 가진다.
   - 여러 기능에 반복 적용되는 품질 보장은 `요건 종류: 비기능`으로 분리한다.

## 현재 카드 1차 판정 초안

```text
REQ-001 이메일 회원 가입: 기능 / application / 원자 요건. REQ-013 화면 슬라이스 병합 완료, 검토중
REQ-013 이메일 회원 가입 화면: 대체됨 / 대체 요건 REQ-001
REQ-002 개인별 할 일 관리: 대체됨 / 대체 요건 REQ-021
REQ-021 개인별 할 일 관리: 기능 / application / 상위 요건. API lifecycle 성과 AC만 소유, 검토중
REQ-022 할 일 생성: 기능 / application / 원자 요건, 검토중
REQ-023 할 일 목록 조회: 기능 / application / 원자 요건, 검토중
REQ-024 할 일 수정: 기능 / application / 원자 요건, 검토중
REQ-025 할 일 삭제: 기능 / application / 원자 요건, 검토중
REQ-026 카테고리 삭제 시 할 일 연결 해제: 기능 / application / 원자 요건, 검토중
REQ-027 할 일 완료 상태 변경: 기능 / application / 원자 요건, 검토중
REQ-003 할 일 카테고리 관리: 대체됨 / 대체 요건 REQ-015
REQ-014 할 일 카테고리 관리 화면: 대체됨 / 대체 요건 REQ-015
REQ-015 카테고리 관리: 기능 / application / 상위 요건. 전체 여정 E2E AC만 소유, 검토중
REQ-016 카테고리 목록 조회: 기능 / application / 원자 요건, 검토중
REQ-017 카테고리 생성: 기능 / application / 원자 요건, 검토중
REQ-018 카테고리 수정: 기능 / application / 원자 요건, 검토중
REQ-019 카테고리 삭제: 기능 / application / 원자 요건, 검토중
REQ-020 신규 사용자 기본 카테고리 제공: 기능 / application / 원자 요건, 검토중
REQ-004 JWT 기반 인증: 정책 / application / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-001, REQ-011, REQ-015, REQ-021
REQ-005 애플리케이션 기본 앱 셸: 기능 / application / 원자 요건. 기존 ID 유지, 승인/BLUE, 제품 영역 platform, 관련 REQ-011
REQ-006 OpenAPI 기반 FE API 계약 검증: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-007, REQ-008
REQ-007 OpenAPI 기반 FE API 클라이언트 생성: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-006, REQ-008
REQ-008 FE API 계약 오류 검사: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-006, REQ-007
REQ-009 Gherkin 시나리오 SCN-* 구조 검사: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-010
REQ-010 통합 하네스 게이트: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-006, REQ-008, REQ-009, REQ-012
REQ-011 이메일·비밀번호 로그인과 로그아웃: 기능 / application / 원자 요건. 기존 ID 유지, 승인/BLUE, 제품 영역 auth, 관련 REQ-001, REQ-004, REQ-005, REQ-021
REQ-012 AC 단위 검증 채널 마커와 하네스 게이트: 하네스 / harness / 원자 요건. 기존 ID 유지, 승인/BLUE, 관련 REQ-010
```

이 초안은 자동 변환 규칙이 아니라 마이그레이션 리뷰 시작점이다. 각 카드의 실제 AC와 의사결정 로그를 보고 canonical 여부를 확정한다.

제목 점검 결과 `REQ-016`과 `REQ-006`은 새 제목 표준에 맞게 정리했다. `REQ-004`는 작업 행위인 `전환`을 제거해 최종 정책 상태인 `JWT 기반 인증`으로 정리했고, `REQ-005`는 구현 기반 표현을 제거해 `애플리케이션 기본 앱 셸`로 정리했다. `REQ-007~REQ-012`는 작업 표현(`표준화`, `강화`, `분리`, `도입`, `적용`)을 제거하고 현재 보장하는 하네스/기능 명세 이름으로 정리했다. `REQ-024`는 `할 일 수정`으로 축소하고, 완료 상태 변경은 신규 `REQ-027 할 일 완료 상태 변경`으로 분리했다.
